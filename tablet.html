Zero training, hundreds of dishes.

Overview

In formal testing, 5 of 5 untrained users completed the full flow.

The Ordering Tablet is a tableside self-ordering device for all-you-can-eat restaurants — usable by any guest with zero training, with tier permissions and round rules enforced by the interface itself instead of a server’s memory.

Constraints: zero training opportunity for users; tier and round rules vary by merchant, so the interface had to be configurable; limited engineering resources demanded strict feature triage.

Key trade‑off: walking away from the poster-style ordering merchants widely hoped for, and concentrating resources on what the target dining format actually needed.

Process

01

The AYCE problem

Hundreds of dishes, on human memory.

In all-you-can-eat hot pot, sushi, and BBQ restaurants, guests order in multiple rounds — a single table can accumulate over a hundred, sometimes several hundred dishes. Relying on servers to enter orders by hand simply doesn’t hold up during peak hours.

And menus in this segment are priced in tiers: what guests can order depends on which tier they paid for. Expecting servers to memorize what each table may order, may not order, and has already ordered is neither realistic nor error-proof.

The core question: how do you let a completely untrained guest smoothly self-order hundreds of dishes — while staying within tier permissions and round-based rules?

02

Who it’s for

The exact opposite of Mobile POS.

The users are dine-in guests seeing this interface for the first time — from children to grandparents, with zero training opportunity and zero willingness to learn. This is the exact opposite design premise from Mobile POS, which serves trained servers: here, the interface must have zero learning cost — anyone should be able to pick it up and use it.

The secondary stakeholders are merchants: they care about operational efficiency and the accurate enforcement of tier permissions and dining rounds.

03

Research

Three merchant demands, three fates.

Before design began, I conducted field observation in real restaurants, studying how guests behave in actual dining environments. Combined with market research and merchant interviews, this converged on the three demands that mattered most:

  • Display ordered items, so guests don’t re-order dishes they’ve forgotten — which wastes food
  • Strictly enforce dining round control
  • A desire for a full-poster, interactive ordering experience, distinct from the traditional image list

These three demands met three different fates: the first two became core features. The third was deliberately dropped after validation.

04

Decision one

Hide it — or visible but locked?

Facing tiered menu permissions, the most straightforward approach is to hide dishes a guest can’t order — the cleanest interface. We didn’t take that path. The final solution: greyed-out + lock badge — higher-tier dishes remain visible, just locked.

  • For guests: they see the full menu, and never wonder “why does that table have a dish I can’t find?”
  • For merchants: a greyed-out higher-tier dish is itself the motivation to upgrade — guests can see it but can’t order it, and the call-server button is exactly that upgrade entry point

The cost is an interface that always looks fuller than one showing only orderable dishes — we accepted the visual density in exchange for the double win of unconfused guests and built-in upgrade motivation.

05

Decision two

Round rules as configuration, not code.

Round rules differ completely across dining formats — hot pot, sushi, and BBQ each restrict differently. So round control couldn’t be fixed logic; it was designed as a merchant-configurable rule system rendered dynamically in the interface — one interface, carrying different round rules in different restaurants.

06

Decision three

The feature we didn’t build.

For the poster-style ordering merchants widely asked for, we built a prototype demo to validate the direction — and ultimately decided not to build it:

  • Engineering cost didn’t match the payoff: high implementation effort against a small base of merchants it would apply to
  • Wrong dining format: poster-style immersion suits high-end dining, while mid-to-low-priced AYCE guests want to order fast, frequently, and in rounds — not admire a beautiful menu

That decision let the team concentrate resources on what the target segment actually needed: tier permissions, round control, and order history.

07

Design system

One kit, many merchants.

As with Mobile POS, I designed a complete UI design system — component library and variant architecture — so restaurants across different merchants and dining formats could assemble their own ordering interface from the same components, instead of redesigning from scratch for every new merchant.

Solution

The guest’s journey, self-served.

01 Menu browsing + left-side categories

Categories stay fixed on the left, so guests switch cuisines without getting lost; dishes are clear image-and-text cards — name, photo, and tier status at a glance.

02 Tier lock badges & upgrade entry

Dishes above the table’s paid tier appear greyed-out with a lock badge — visible but not orderable; once an upgrade completes via the call-server button, locked dishes unlock instantly.

03 Cart

When a whole table orders together, the cart is the confirmation zone before submitting: quantities and options at a glance, preventing mis-orders.

04 Order history

Guests can check what the table has already ordered at any time — preventing duplicate orders at the source.

05 Call server

Self-service doesn’t mean no service. One tap summons a server for tier upgrades, special requests, or questions — reserving human service for the moments that genuinely need a human.

06 Round control

Merchant-configured round rules render dynamically — guests know exactly what they can still order this round, reducing rule disputes between guests and servers.

Impact

Tested, and only small fixes needed.

After launch, I led formal user testing: five participants who had never seen the product walked through the complete self-ordering flow — five being the standard sample size for qualitative usability testing, enough to surface the vast majority of issues.

All five untrained users completed the main flow — browse → order → view ordered items → call server — without assistance. Only minor refinements followed, with no structural usability issues found. For a product built for untrained users, “tested, and only small fixes needed” is itself the evidence that the design goal was met.

Reflection

The most important lesson came from a feature we didn’t build.

Merchants widely hoped for a full-poster ordering experience; we prototyped it, then deliberately walked away. It’s all called “restaurant ordering,” but a hot pot chain, a high-end omakase, and a fast-food counter need completely different things.

That experience built a habit I’ve kept ever since: whenever a restaurant-related design request comes in, I ask about the dining format first — it determines which features are worth building, and which appealing-sounding requests deserve a firm no. Designing round control as a configurable rule system rather than fixed logic is the same principle applied in the positive direction: one interface, serving many dining formats.
